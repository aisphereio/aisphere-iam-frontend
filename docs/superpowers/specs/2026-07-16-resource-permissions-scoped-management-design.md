# 资源权限按资源类型收敛设计

## 1. 背景与问题

当前“资源权限”页面把项目和不同业务域资源混在一棵资源树里。用户从某个项目点击“添加成员”后，页面跳转到全局“访问分配”，但没有保留项目上下文，必须重新选择资源类型和具体资源。

这造成三个直接问题：

- 用户无法确定当前授权究竟作用于项目、空间还是具体资源。
- 从资源详情进入授权流程后重复选择资源，操作链路割裂。
- 项目、Skill、Git 仓库、运行环境等异构资源混排，列表规模和角色语义都难以理解。

## 2. 设计目标

- “资源权限”继续管理 IAM 中所有可授权资源，而不是只管理项目。
- 进入页面默认管理“项目”，用户可切换到其他资源类型。
- 任意成员操作都绑定当前选中的资源，添加成员时不再重新选择资源。
- 角色选项只显示当前资源类型支持的角色。
- 添加或撤销后，当前资源的成员和有效权限原地刷新。
- 页面始终明确展示当前操作范围，避免误授权。

## 3. 方案比较

### 方案 A：资源类型筛选 + 同类型资源列表 + 详情面板（采用）

顶部选择资源类型，左侧只显示该类型的资源实例，右侧显示当前资源的成员与权限。“添加成员”在当前页面打开侧边抽屉或对话框，资源上下文锁定。

优点：认知范围小、上下文稳定、可以复用所有资源类型和角色目录接口。缺点：跨资源类型切换需要一次下拉操作。

### 方案 B：保留完整异构资源树（不采用）

按组织、项目、空间、具体资源展示完整层级树，并在树节点上管理权限。

优点：可以看到完整层级。缺点：15 类资源混合后树过深，搜索、懒加载、继承说明和移动端交互都复杂；当前实现也没有正确表达多层父子关系。

### 方案 C：每种资源类型独立管理页面（不采用）

为项目、Skill、Git 仓库、运行环境分别建立权限页面。

优点：每页可以高度定制。缺点：页面和导航数量快速膨胀，授权交互重复，后续新增资源类型需要持续增加页面。

## 4. 页面结构

```text
资源权限
管理资源成员、角色以及继承权限

资源类型 [项目 v]                 搜索项目 [________]

+----------------------+  +-------------------------------------+
| 项目                 |  | AI 助手平台          [添加成员]    |
|                      |  | 项目 · project:project-id           |
| > AI 助手平台        |  +-------------------------------------+
|   数据平台           |  | [直接成员] [有效权限]               |
|   模型中心           |  |                                     |
|                      |  | 张三     项目管理员      [移除]     |
|                      |  | 研发组   项目开发者      [移除]     |
+----------------------+  +-------------------------------------+
```

### 4.1 资源类型选择

- 数据来自资源类型目录，只显示 `grantable != false` 的类型。
- 默认值为 `project`；若目录没有 `project`，选择第一个可授权类型。
- 切换类型时清空当前资源选择、搜索和详情查询状态。
- 类型名称使用业务展示名，不直接暴露内部英文标识。

### 4.2 资源列表

- 只请求并显示当前资源类型的实例。
- 项目使用项目目录作为数据源；其他类型使用资源目录并传入 `type` 过滤。
- 列表项显示名称和必要的上级路径，避免同名资源无法区分。
- 加载完成后默认选择第一个资源；用户手动选择优先。
- 空状态明确说明该类型暂无资源，并提供切换资源类型的指引。

### 4.3 资源详情

- 标题显示资源名称、类型和稳定 ID。
- “直接成员”显示直接授予当前资源的用户或用户组。
- “有效权限”显示直接授权与继承结果，并明确来源。
- 当前没有选中资源时，不允许添加成员。

## 5. 添加成员流程

“添加成员”在资源详情内打开成员分配面板，不跳转到全局授权页面。

面板固定展示但不允许修改：

- 当前资源类型
- 当前资源名称
- 当前资源 ID

用户只需填写：

1. 对象类型：人员或用户组。
2. 具体人员或用户组。
3. 当前资源类型支持的角色。
4. 可选的到期时间和授权原因。

提交沿用现有 Grant API：

```json
{
  "resource": { "type": "project", "id": "project-id" },
  "role_key": "developer",
  "subject": { "type": "user", "id": "user-id" },
  "source": "iam_console"
}
```

提交成功后关闭面板，刷新当前资源的直接成员、有效权限和授权列表，不改变当前类型与资源选择。

## 6. 状态与 URL

- 页面状态至少包含 `resourceType`、`resourceId`、搜索词和详情页签。
- `resourceType` 与 `resourceId` 写入查询参数，刷新页面后保留上下文。
- 无效或已删除的资源 ID 自动回退到当前类型的第一个资源。
- 从其他页面进入资源权限时，允许通过查询参数预选资源。

## 7. 数据流与组件边界

- `ResourcePermissions` 负责资源类型、资源实例选择和详情查询。
- 新的 `ResourceMemberDialog` 只负责当前资源的成员分配，不拥有资源选择权。
- 全局 `GrantEditor` 继续作为高级批量/跨资源授权入口，但不再承担资源详情中的“添加成员”。
- 角色模板按 `resourceType` 过滤；用户和用户组沿用现有目录接口。
- 后端接口和 proto 不需要为本次交互调整，现有 Grant、AccessQuery、资源目录与角色目录接口足够支撑。

## 8. 错误与空状态

- 资源类型加载失败：在筛选区显示错误和“重新加载”。
- 资源列表加载失败：保留资源类型选择，列表区显示重试按钮。
- 权限加载失败：保留当前资源，详情区显示错误和重试按钮。
- 成员分配失败：面板保持打开并保留输入，显示后端错误信息。
- 撤销失败：不移除现有成员行，显示错误提示。
- 角色为空：禁用提交，说明当前资源类型尚未配置可分配角色。

## 9. 可用性与可访问性

- 资源类型下拉、资源列表和成员面板支持键盘操作与可见焦点。
- 列表选择状态同时使用背景、图标和 `aria-selected`，不只依赖颜色。
- 危险操作继续使用确认对话框。
- 小屏幕下资源列表和详情纵向排列；成员面板使用适合移动端的宽度。
- 不新增装饰性动画，只保留现有页面切换和面板过渡。

## 10. 验收标准

- 首次进入资源权限页默认展示项目列表。
- 用户可以切换任意可授权资源类型，并只看到该类型资源。
- 选择资源后，详情只展示该资源的直接成员与有效权限。
- 从详情点击“添加成员”不会离开资源权限页，也不能重新选择资源。
- 成员面板只提供与当前资源类型匹配的角色。
- 添加或撤销成员后，当前资源详情自动刷新且上下文不丢失。
- 刷新浏览器后能恢复合法的资源类型和资源选择。
- 加载、空数据、接口失败和无角色配置都有明确可操作提示。

## 11. 测试边界

组件级公共行为测试覆盖：

- 默认选择项目并加载项目列表。
- 切换资源类型后按类型加载资源并清空旧选择。
- 选中资源后以正确的 `{type, id}` 查询权限。
- 添加成员面板锁定当前资源，按当前类型过滤角色。
- 成功授权后刷新当前资源权限且保持选择。
- URL 参数恢复和无效资源回退。
- 资源、权限和授权失败状态允许重试。

## 12. 非目标

- 本次不重新定义项目或其他资源类型的权限模型。
- 本次不修改 SpiceDB schema、Grant proto 或 AccessQuery proto。
- 本次不实现跨多个资源的批量授权。
- 本次不把所有业务资源重新组织为完整可展开层级树。
